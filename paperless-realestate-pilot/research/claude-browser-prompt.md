# Промпт для Claude (браузер) — поиск решения для индексации документов без дублирования

Скопируй всё ниже и вставь в claude.ai:

---

## Контекст

Я разворачиваю систему управления документами для клиента в сфере недвижимости. У клиента:

- Windows-сервер с SMB-шарой (сотни гигабайт документов: PDF, сканы, Word, Excel)
- Пользователи работают с шарой через проводник Windows
- Главная потребность — **полнотекстовый поиск** по содержимому документов на шаре
- Документы должны оставаться на шаре, структура папок не должна меняться
- Клиенту НЕ нужна полноценная DMS (теги, workflow, классификация) — нужен именно поиск

## Что мы уже сделали

Изначально развернули **Paperless-ngx** (Docker Compose: Paperless + PostgreSQL + Redis + Tika + Gotenberg). Проблема: Paperless архитектурно копирует каждый документ к себе в media-директорию (оригинал + PDF/A-архив). Это удваивает-утраивает потребление диска. Нет режима "только индексация" — Paperless забирает файл из consume-директории и хранит его как свой.

## Что мы исследовали

Провели анализ альтернатив:

1. **Recoll + recoll-webui** — лучший кандидат. Индексирует файлы на месте (Xapian-индекс, 4-30% от размера документов). OCR через Tesseract, поддержка русского. Docker-образы есть. Минус: веб-интерфейс устаревший и базовый.

2. **Elasticsearch + FSCrawler + Kibana** — production-grade. FSCrawler сканирует FS и индексирует в ES без копирования. Мощный поиск. Минус: тяжёлый стек (ES 4+ ГБ RAM), Kibana сложен для обычных пользователей, FSCrawler не обновлялся стабильно с 2022.

3. **Manticore Search** — лучшая поддержка русского языка (российская команда, встроенный лемматайзер). Но нет файлового краулера и нет веб-интерфейса — нужно всё писать самим.

4. **Ambar, Open Semantic Search** — мёртвые проекты (2020-2021).

5. **Meilisearch, Teedy** — не подходят (нет in-place индексации).

## Идея с форком Paperless-ngx

Рассматриваем вариант модификации Paperless-ngx чтобы он работал в режиме "search-only":
- Consumer обрабатывает документ (OCR, извлечение текста, построение индекса), но не копирует оригинал в свой media — вместо этого хранит ссылку (путь) на файл на шаре
- При скачивании/просмотре документа — отдаёт файл по этому пути
- Thumbnail и поисковый индекс хранятся локально (это мелочь по объёму)
- Sanity checker не ругается на отсутствие локальных оригиналов

Плюсы: получаем готовый UI Paperless (красивый, с превью, фильтрами, полнотекстовым поиском) без хранения дублей.
Минусы: нужно глубоко лезть в Django-код Paperless (~50k строк), потеря совместимости с upstream.
